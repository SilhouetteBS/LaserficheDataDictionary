# Reporting SQL Review Workflow

The Reporting guide has two different kinds of community SQL entries:

- Curated scripts: reusable, portable read-only reporting patterns stored under `reporting/<product>/`.
- Schema-matched candidates: generated source candidates from Laserfiche Answers posts whose referenced object names exist in at least one imported schema version.

## Generated Candidate Data

`src/data/generatedReportingCandidates.js` is generated by:

```powershell
npm run verify:answers-schema
```

Do not edit that file by hand. Update the source processed queue, reviewed-post ledger, or verification tool, then regenerate it.

The generator also writes:

```text
docs/answers-sql-schema-verification-2026-07-01.md
```

That report is the public audit trail for the current schema-verification pass.

## Promotion Flow

1. Search or seed Laserfiche Answers posts with `npm run research:answers-sql`.
2. Document processed rows in the public `docs/answers-sql-processed-*.md` files.
3. Run `npm run verify:answers-schema`.
4. Review schema-matched candidates in the Reporting guide.
5. Promote useful candidates into curated scripts under `reporting/<product>/`.
6. Keep candidate entries tagged `Needs review` until they are rewritten, reviewed, and validated.

## What Schema Matched Means

`Schema matched` means the referenced object names were found in imported Data Dictionary schemas for the selected product. It does not mean:

- the SQL was live tested
- the SQL is production ready
- the query is performant
- the query is supportable as written
- the result columns were validated against real customer data

Users should treat these entries as source research and validate in a test environment.

## No-Reference Candidates

Rows marked `No referenced objects captured` are kept in the schema-verification report, but they are not published as Reporting entries. Common reasons include:

- CSS or JavaScript snippets were extracted instead of SQL
- the post discussed an external/customer database
- the SQL was described in prose instead of code
- the post was troubleshooting context rather than a reusable query
- the snippet did not include recognizable Laserfiche product object names

These rows can be revisited later if a better extractor or manual review captures usable object references.

## Bundle Size

Generated candidates are lazy-loaded by the Reporting guide so the main app bundle stays below the performance budget. `npm run validate` includes `npm run audit:performance`, which checks the JavaScript and CSS budgets after build.
